Le code est lu bien plus souvent qu'il n'est écrit. Écrire **clair**, c'est se respecter soi-même et respecter son équipe.

## 1. Des noms qui parlent
`x`, `data`, `truc` ne veulent rien dire. `prixTotal`, `listeUtilisateurs`, `estConnecte` : on comprend **sans commentaire**.

## 2. Une fonction = une tâche
Une fonction qui fait dix choses est impossible à comprendre. Découpe : chaque fonction fait **une** chose, bien.

## 3. Évite la duplication
Si tu copies-colles le même code, transforme-le en fonction réutilisable. « Don't Repeat Yourself ».

## 4. Des commentaires utiles
Ne commente pas l'évident. Commente le **pourquoi**, pas le **quoi**. Le bon code se lit presque comme une phrase.

## 5. De la cohérence
Même style, même façon de nommer, partout. La cohérence rend le code prévisible et agréable.

## Pourquoi ça compte
- Tu **retrouves** ton chemin dans tes vieux projets
- Tu travailles **en équipe** sans friction
- Les recruteurs voient un **professionnel**, pas un bricoleur

> Le code propre n'est pas un luxe d'esthète. C'est la différence entre un développeur qu'on garde et un dont on se passe.
